

# SCADA 实时数据平台 — 技术方案文档

---

## 一、项目概述

本项目是一套基于 Java 的 SCADA（数据采集与监控）实时数据平台，面向工业物联网场景，实现对现场设备数据的实时采集、处理、存储、报警、任务调度和前端可视化推送。

系统核心目标：
- 支撑 100 个以上网关的数据并发接入
- 前端实时数据 100ms 级别更新
- 支撑 1000 人同时在线访问
- 数据处理严格保序，不因并发丢失时序关系
- 后端处理引擎卡顿不影响前端展示

### 2.1 sys_pages 页面表设计

#### 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL PK | 自增主键 |
| page_key | VARCHAR(64) UNIQUE | 页面唯一标识（前端路由用，如 `overview-board`） |
| name | VARCHAR(128) NOT NULL | 页面名称 |
| description | TEXT | 页面描述 |
| status | VARCHAR(16) NOT NULL DEFAULT 'draft' | 页面状态：draft / enabled / disabled |
| schema_json | JSONB NOT NULL | 完整的页面 Schema（组件树、变量、脚本等） |
| thumbnail | TEXT | 缩略图 Base64 或 URL，列表展示用 |
| version | INT NOT NULL DEFAULT 1 | 乐观锁版本号，防止并发覆盖 |
| created_by | BIGINT | 创建人 user_id |
| updated_by | BIGINT | 最后修改人 user_id |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 创建时间 |
| updated_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 最后修改时间 |

#### 索引

```sql
CREATE INDEX idx_pages_status ON sys_pages(status);
CREATE INDEX idx_pages_updated ON sys_pages(updated_at DESC);
-- JSONB 内组件类型查询（按需）
CREATE INDEX idx_pages_schema_gin ON sys_pages USING GIN(schema_json jsonb_path_ops);
```

#### 为什么用 JSONB 而非拆表

页面 Schema 是前端拖拽编辑器的产物，结构为深层嵌套的组件树（root → children → children …），包含任意层级的组件属性、变量绑定和脚本代码。拆成关系表会导致：
- 递归查询组件树开销大
- 每次保存需要删除重建整棵子树
- Schema 结构随前端组件升级频繁变化，关系表改表成本高

JSONB 的优势：
- PostgreSQL JSONB 支持索引、部分更新（jsonb_set）、路径查询
- 整体读写一次 IO，与前端 JSON 结构完全对齐
- 单个页面 Schema 通常 50KB~500KB，远在 PostgreSQL JSONB 1GB 上限之内

#### 缓存策略

页面不需要独立的应用层缓存，理由：

1. **访问频率低**：页面 Schema 只在两个时机读取——编辑器打开时加载一次、预览/运行时加载一次。不像变量数据每秒数千次读取
2. **PostgreSQL 自带缓存**：热点页面会驻留在 PostgreSQL 的 shared_buffers 中，重复读取命中率极高
3. **前端已缓存**：编辑器打开后整个 Schema 加载到前端内存（Zustand Store），整个编辑会话期间不再请求后端
4. **写入频率低**：用户保存操作通常几十秒到几分钟一次

如果未来出现万人同时打开同一页面的极端场景，可在应用层加一层 `ConcurrentHashMap<String, PageSchema>` 或 Caffeine 缓存，以 page_key 为 key、设置 5 分钟过期即可。初期不需要。

#### 并发编辑保护

使用乐观锁防止两人同时编辑覆盖：

```sql
UPDATE sys_pages
SET schema_json = :newSchema, version = version + 1, updated_at = now()
WHERE id = :id AND version = :expectedVersion;
```

受影响行数为 0 时返回冲突提示，前端提示用户刷新后重试。

### 2.2 页面编辑锁设计（悲观锁 + 心跳续期）

编辑锁与乐观锁是两层保护，职责不同：
- **编辑锁**（悲观锁）：防止两人同时打开同一页面的编辑器，从 UI 层面阻止冲突
- **乐观锁**（version 字段）：兜底保护，万一编辑锁失效，保存时仍能检测到冲突

#### sys_page_locks 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| page_id | BIGINT PK | 页面 ID，同时是主键（一页最多一把锁） |
| locked_by | BIGINT NOT NULL | 持有锁的用户 user_id |
| locked_by_name | VARCHAR(64) NOT NULL | 持有锁的用户显示名（冗余，供前端直接展示） |
| locked_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 加锁时间 |
| expires_at | TIMESTAMPTZ NOT NULL | 锁过期时间，默认 locked_at + 5 分钟 |

#### 工作流程

```
用户 A 点击「编辑」
  ↓
后端尝试加锁：
  INSERT INTO sys_page_locks(page_id, locked_by, locked_by_name, locked_at, expires_at)
  VALUES(:pageId, :userId, :userName, now(), now() + interval '5 minutes')
  ON CONFLICT(page_id) DO NOTHING;
  ↓
  插入成功 → 返回页面 Schema，进入编辑器
  插入失败 → 查询当前锁持有者：
    if expires_at < now() → 锁已过期，强制替换（见下方过期处理）
    else → 返回 409 Conflict + { lockedBy: "张三", lockedAt: "10:23:15" }
         → 前端提示「该页面正在被 张三 编辑中，请稍后再试」
```

#### 心跳续期

前端编辑器每 **2 分钟**发一次心跳请求：

```
PUT /api/pages/:pageId/lock/renew
```

后端收到后：

```sql
UPDATE sys_page_locks
SET expires_at = now() + interval '5 minutes'
WHERE page_id = :pageId AND locked_by = :userId;
```

如果用户正常编辑，锁永远不会过期。如果用户浏览器崩溃、断网、直接关机，心跳停止，5 分钟后锁自动失效。

#### 释放锁

三种释放方式：

| 触发时机 | 释放方式 |
|---------|---------|
| 用户点击「保存」或「退出编辑」 | 主动 DELETE |
| 用户关闭浏览器标签页 | 前端 `beforeunload` 事件中发送释放请求（navigator.sendBeacon） |
| 用户崩溃/断网 | 心跳停止，5 分钟后过期自动失效 |

主动释放 SQL：

```sql
DELETE FROM sys_page_locks WHERE page_id = :pageId AND locked_by = :userId;
```

#### 过期锁处理

其他用户尝试加锁时发现已有锁但已过期：

```sql
DELETE FROM sys_page_locks WHERE page_id = :pageId AND expires_at < now();
-- 然后重新 INSERT
```

#### 管理员强制解锁

管理员角色可强制释放他人的锁（无需等过期）：

```
DELETE /api/pages/:pageId/lock?force=true
```

仅限拥有 `page:force_unlock` 权限的角色。

### 2.3 页面权限设计（RBAC 解耦）

权限模型采用经典 RBAC，把「谁能访问哪个页面」的关系从页面表和用户表中彻底剥离，全部通过角色中转。

#### 关系模型

```
sys_users ──（多对多）──▶ sys_roles ──（多对多）──▶ sys_pages
              via                       via
         sys_user_roles            sys_role_pages
```

页面表不存权限信息，用户表不存页面信息，两者完全解耦。

#### sys_roles 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL PK | 自增主键 |
| role_key | VARCHAR(64) UNIQUE NOT NULL | 角色标识（如 `admin`、`operator`、`viewer`） |
| role_name | VARCHAR(128) NOT NULL | 角色显示名（如「系统管理员」「操作员」「只读用户」） |
| description | TEXT | 角色描述 |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 创建时间 |

#### sys_user_roles 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | BIGINT NOT NULL | 用户 ID |
| role_id | BIGINT NOT NULL | 角色 ID |
| PRIMARY KEY | (user_id, role_id) | 联合主键 |

#### sys_role_pages 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| role_id | BIGINT NOT NULL | 角色 ID |
| page_id | BIGINT NOT NULL | 页面 ID |
| access_level | VARCHAR(16) NOT NULL DEFAULT 'view' | 权限级别：`view`（只读预览）/ `edit`（可编辑） |
| PRIMARY KEY | (role_id, page_id) | 联合主键 |

#### 权限判断流程

```
用户请求访问页面
  ↓
查询用户所有角色：
  SELECT role_id FROM sys_user_roles WHERE user_id = :userId
  ↓
查询角色对该页面的最高权限：
  SELECT MAX(access_level) FROM sys_role_pages
  WHERE role_id IN (:roleIds) AND page_id = :pageId
  ↓
  无记录       → 403 无权访问
  view         → 允许预览，编辑按钮灰置
  edit         → 允许预览 + 编辑（仍需获取编辑锁）
```

权限合并规则：**取最高权限**。用户有多个角色时，任一角色授予了 `edit` 则可编辑。

#### 内置角色

| role_key | 说明 | 默认页面权限 |
|----------|------|------------|
| admin | 系统管理员 | 所有页面 edit + 强制解锁 + 管理用户/角色 |
| designer | 页面设计师 | 被分配页面 edit |
| operator | 操作员 | 被分配页面 view（预览运行页面，可下发控制指令） |
| viewer | 只读用户 | 被分配页面 view |

#### 解耦好处

1. **新增页面**：不需要改用户表，只需在 sys_role_pages 给相关角色授权
2. **新增角色**：不需要改页面表，只需创建角色并分配用户和页面
3. **调整权限**：一行 SQL（INSERT/UPDATE/DELETE sys_role_pages），无需触碰业务表
4. **查询高效**：用户登录时一次查出所有角色和页面权限，缓存到 JWT 或会话中
5. **与设备权限统一**：sys_permissions 原有的设备/变量权限也走角色，两套权限共享同一个角色体系

#### 页面列表接口的权限过滤

```sql
-- 用户能看到的页面列表（自动过滤无权限的页面）
SELECT p.*, rp.access_level
FROM sys_pages p
JOIN sys_role_pages rp ON rp.page_id = p.id
JOIN sys_user_roles ur ON ur.role_id = rp.role_id
WHERE ur.user_id = :userId
GROUP BY p.id
ORDER BY p.updated_at DESC;
```

admin 角色跳过过滤，直接返回所有页面。

### 4.2 三级解耦保证前端不卡

系统分为三个独立层级，任何一层卡顿不影响其他层：

```
第一级：采集层（MQTT 接收 + 分区队列）
  - 只做收消息和分发，极快
  - 绝不执行任何业务逻辑

第二级：处理层（变量引擎 + 任务引擎 + 脚本执行）
  - 所有重逻辑都在这里
  - 可能因为脚本执行、数据库写入等原因变慢
  - 变慢只会导致数据延迟，不会影响前端连接

第三级：推送层（WebSocket 推送服务）
  - 独立运行，只从内存缓存读最新值
  - 处理层卡了，推送层照样推上一次的有效值
  - 前端不会卡顿、不会断连、不会白屏
```

## 五、核心模块详细设计

### 5.1 MQTT 接收层

职责：接收所有网关的 MQTT 消息，按网关分区投入有序队列。

技术细节：
- 使用单个 MQTT 连接，通配符订阅所有网关 topic，如 device/+/data
- MQTT 回调函数中不做任何业务处理
- 回调中只做一件事：计算分区号，投入对应队列
- 分区算法：partition = hash(gateway_id) % N
- 分区数 N 建议 8 到 16，100 个网关平均每个分区承担 6 到 12 个网关

### 5.2 分区有序队列

职责：保证同一个网关的数据严格按到达顺序处理。

核心设计：
- N 个独立的 BlockingQueue，每个队列对应一个固定的消费者线程
- 同一个 gateway_id 的消息永远进同一个队列
- 每个队列只有一个 Worker 线程消费，严格 FIFO
- 不同队列之间完全并行，互不影响
- 系统总吞吐量 = 单队列吞吐量 x N

为什么不能用一个大线程池随机消费：
- 随机消费会导致同一网关的两条消息被不同线程并发处理
- 后到的消息可能先处理完，导致数据乱序
- 乱序数据会造成报警误报、历史曲线毛刺、条件判断错误

### 5.3 变量处理引擎

职责：对每个变量的原始数据进行完整的处理链。

内存数据结构：

```
ConcurrentHashMap<Long, VariableRuntime> variableCache
  key:   var_id
  value: 变量运行时状态，包含：
         - config: 变量配置（从 sys_variables 加载）
         - lastValue: 上次的值
         - lastQuality: 上次的质量戳（192=Good, 0=Bad, 64=Uncertain）
         - lastStoreTime: 上次存储时间
         - alarmState: 当前报警状态（正常/H/HH/L/LL）

ConcurrentHashMap<Long, List<TaskRuntime>> varToTasks
  key:   var_id
  value: 关联的任务列表（倒排索引）
```

单条数据处理流程（在分区 Worker 线程内同步串行执行）：

```
第 0 步：协议适配（在投入分区队列前完成）
  由 GatewayParser 将原始报文解析为 List<RawPoint>
  数字键翻译、共享值合并、质量戳提取都在这一步完成
  后续步骤只看 RawPoint 统一结构，不关心原始报文格式

第 1 步：变量匹配
  根据 RawPoint.varTag 匹配 sys_variables 中的变量配置
  匹配不到的 tag 丢弃并记录告警日志

第 2 步：质量戳检查
  if RawPoint.quality = Bad (0) → 更新质量状态到缓存，跳过后续处理
  if RawPoint.quality = Uncertain (64) → 正常处理但标记为不确定
  if RawPoint.quality = Good (192) → 正常处理

第 3 步：量程转换
  工程值 = 原始值 * scale_factor + offset_val

第 4 步：计算点派生
  如果 source_type = 1（计算点）
  使用 Aviator 引擎执行 calc_rule
  可引用其他变量的当前值

第 5 步：数据改变脚本（如果有关联的 task_type=2 任务）
  在当前线程内同步执行 change_type 判断
  判断类型包括：
    ANY:           任何变化
    INCREASE:      新值 > 旧值
    DECREASE:      新值 < 旧值
    THRESHOLD:     |新值 - 旧值| > change_threshold
    FALSE_TO_TRUE: 从 0 变为 1
    TRUE_TO_FALSE: 从 1 变为 0

第 6 步：条件事件评估（如果有关联的 task_type=3 任务）
  在当前线程内同步执行 condition_expr 评估
  使用 Aviator 解析表达式
  注入当前所有相关变量的最新值
  表达式为 true 时标记需要触发 action

第 7 步：存储策略判断
  store_mode = 0（不存）: 跳过
  store_mode = 1（变化）: |新值 - 旧值| > store_deadband 时存储
  store_mode = 2（定时）: 当前时间 - lastStoreTime > store_cycle 时存储
  store_mode = 3（混合）: 以上两个条件任一满足即存储
  附加项：store_on_startup = 1 时，服务启动后首次收到该变量数据无条件存储一次

第 8 步：报警检查
  如果 alarm_enable = 1
  按优先级检查：新值 > limit_hh → HH 报警
                新值 > limit_h  → H 报警
                新值 < limit_ll → LL 报警
                新值 < limit_l  → L 报警
  带死区防抖：报警恢复时，值必须回到阈值 - deadband 以内才算恢复
  报警状态变化时写入 sys_alarm_records

第 9 步：更新内存缓存
  variableCache.put(varId, newRuntime)
  这是推送服务的数据源

第 10 步：异步投递
  需要存储的数据 → 投入存储队列（异步落库）
  需要执行的 action → 封装 ActionEvent（带时间戳和序号）→ 投入 Action 队列
  变化的数据 → 通知推送服务
```

### 5.4 任务引擎

职责：管理三种类型的任务触发和 Action 执行。

#### 定时任务（task_type = 1）

- 使用 ScheduledThreadPoolExecutor 管理
- 支持 cron 表达式（通过 CronExpression 解析）和固定间隔（interval_sec）
- cron_expr 优先于 interval_sec
- 独立线程池，与数据处理流水线无关
- 到达触发时间直接执行 action

#### 数据改变任务（task_type = 2）

- 在变量处理引擎的第 5 步中同步触发
- 不异步投递，保证判断时值的顺序是正确的
- 判断通过后，action 执行异步投递

#### 条件事件任务（task_type = 3）

- 在变量处理引擎的第 6 步中同步触发
- condition_expr 的解析和评估在当前线程内完成
- 启动时预解析所有条件表达式，提取引用的变量名
- 建立倒排索引：var_name → 关联的条件任务列表
- 任何一个被引用的变量发生变化，就触发该条件表达式的重新评估

#### Action 执行器

- 独立线程池，与数据处理分离
- 同一个变量触发的多个 action 通过分区队列保序
- 每个 action 有超时控制，单次执行最长不超过 5 秒
- 执行结果写回 sys_tasks 的 last_run_time 和 last_run_status

五种动作类型：

| action_type | 说明 | 执行方式 |
|-------------|------|---------|
| 1 = HTTP 请求 | 调用外部接口 | HttpClient 异步发送 |
| 2 = 设备回写 | 向设备下发控制指令 | 通过 WriteDispatcher 统一回写 |
| 3 = 数据库操作 | 执行 SQL | JdbcTemplate |
| 4 = 执行脚本 | 运行用户 Python/JS 脚本 | GraalVM 沙箱 |
| 5 = 写日志 | 记录日志 | Logger 写入 |
| 6 = MQTT 发布 | 向任意 topic 发消息（非设备回写） | MQTT Client 直接发布 |

### 5.5 脚本执行沙箱

职责：安全执行用户编写的 Python 和 JavaScript 脚本。

技术方案：
- 使用 GraalVM Polyglot 引擎，同时支持 JavaScript 和 Python
- 每次执行在独立沙箱上下文中运行
- 注入的上下文变量：当前值（value）、旧值（oldValue）、变量名（tag）、时间戳（timestamp）
- 注入的上下文函数：writeTag(tag, value) —— 内部调用 WriteDispatcher 统一回写，来源标记为 SCRIPT

安全限制：
- 单次执行超时上限：100ms，超时直接终止
- 禁止访问文件系统
- 禁止发起网络请求（除非在 action_config 中显式开放）
- 禁止访问 Java 宿主类
- 独立线程池运行，不占用变量处理线程

### 5.6 WebSocket 推送服务

职责：将变量最新值实时推送到前端，同时接收前端的控制指令。

#### 数据推送（服务端 → 前端）

内存数据结构：

```
ConcurrentHashMap<String, Set<Long>> sessionSubscriptions
  key:   WebSocket sessionId
  value: 该用户订阅的变量 ID 集合

ConcurrentHashMap<Long, Set<String>> varToSessions
  key:   var_id
  value: 订阅该变量的 sessionId 集合（倒排索引）
```

推送策略：
- 变量值变化时，查倒排索引找到所有订阅者
- 攒批推送：100ms 内的变化攒成一批，一次性推送给前端
- 只推变化的变量，不推全量
- 每条推送消息携带时间戳 ts

推送消息格式：

```
{
  "type": "data",
  "tag": "temperature_01",
  "var_id": 15,
  "value": 36.5,
  "quality": 192,
  "alarm_level": "H",
  "ts": 1712567890123
}
```

quality 字段使用 OPC 质量码整数值：192=Good、0=Bad、64=Uncertain。前端根据 quality 决定显示样式（Bad 时变量值灰置并标记感叹号）。

#### 控制指令（前端 → 服务端）

指令消息格式：

```
{
  "type": "command",
  "tag": "valve_01",
  "action": "open",
  "operator": "user_123",
  "ts": 1712567890456
}
```

服务端收到后：
- 反序列化为 WriteCommand
- 交给 WriteDispatcher.dispatch(command, WS) 统一处理（权限校验 → 协议编码 → MQTT 发布 → 等待回执 → 审计日志）
- 拿到 WriteResult 后组装 command_ack 推给前端

回执消息格式：

```
{
  "type": "command_ack",
  "tag": "valve_01",
  "status": "success",
  "ts": 1712567890789
}
```

#### 前端乱序兜底

前端维护每个变量的最新时间戳，收到数据时判断：
- 如果新数据的 ts 小于等于当前展示的 ts，直接丢弃
- 保证前端展示的始终是最新数据

#### 断线处理

- WebSocket 断开后自动重连
- 重连成功后重新订阅变量
- 断连期间前端展示最后一次有效数据，不白屏

### 5.7 存储服务

职责：将需要持久化的数据异步写入数据库。

#### 时序数据存储

- 变量处理引擎根据 store_mode 判断后，将需要存储的数据投入存储队列
- 存储服务从队列消费，批量写入时序数据库
- 批量写入策略：每 500 条或每 1 秒写一批，取先到者
- 使用 PostgreSQL 分区表存储时序数据，按时间范围自动分区；如需更高写入性能可启用 TimescaleDB 扩展

#### 报警记录存储

- 报警状态变化时直接写入 sys_alarm_records
- 记录字段包括：报警产生时间、报警等级、报警值、确认时间、恢复时间

### 5.8 倒排索引设计

系统中有三处使用倒排索引思想：

#### 索引一：变量 → 任务映射

```
var_id → [task1, task2, task3, ...]
```

用途：变量值变化时，O(1) 找到所有关联的数据改变任务和条件事件任务。
构建时机：服务启动时从 sys_tasks 表加载，运行时任务配置变更时热更新。

#### 索引二：变量名 → 条件事件任务映射

```
var_name → [condition_task1, condition_task2, ...]
```

用途：条件表达式中引用了多个变量，任何一个变量变化都需要触发表达式重新评估。
构建方式：启动时解析所有 condition_expr，提取变量名，建立映射。

#### 索引三：变量 → WebSocket 会话映射

```
var_id → [session1, session2, session3, ...]
```

用途：变量值变化时，O(1) 找到所有订阅了该变量的前端连接。
维护方式：用户订阅/取消订阅时动态更新。

### 5.9 网关协议适配层

职责：屏蔽不同网关/设备的 MQTT 报文格式差异，为变量处理引擎提供统一的数据结构，为回写调度器提供统一的指令编码能力。

#### 问题背景

不同厂商的网关上报数据格式完全不同，典型差异如下：

**格式差异**：

```
厂商 A（扁平 JSON）:
  { "temp": 36.5, "pressure": 101.3, "ts": 1712567890 }

厂商 B（嵌套 JSON）:
  { "device": "pump_01", "data": { "temperature": 36.5 }, "timestamp": "2026-04-08T10:00:00Z" }

厂商 C（数组格式）:
  { "points": [{ "tag": "temp", "val": 36.5 }, { "tag": "press", "val": 101.3 }] }

厂商 D（二进制 + Base64）:
  { "raw": "AQIDBA==", "encoding": "base64" }
```

**推送模式差异**：

```
模式一（每次全量）:
  每次推送所有变量的当前值，简单但带宽大

模式二（首次全量 + 后续增量）:
  第一次连接时推送所有变量值
  后续只推送发生变化的变量
  没推的变量 = 值没变，沿用上次

模式三（纯增量）:
  永远只推变化的变量，需要系统自己维护最后一次全量快照
```

**数据结构差异（数字键 / 共享值等奇怪规则）**：

```
厂商 E（数字键约定）:
  {
    "vars": [
      { "1": "temperature_01", "2": 192, "3": 36.5 },
      { "1": "pressure_01", "2": 192, "3": 101.3 }
    ]
  }
  其中 "1" = 变量名，"2" = 质量戳（OPC 质量码，192=Good），"3" = 值

厂商 F（共享值 + 差异值）:
  {
    "shared": { "2": 192, "4": 1712567890 },
    "vars": [
      { "1": "temp", "3": 36.5 },
      { "1": "press", "3": 101.3 },
      { "1": "flow", "3": 22.8 }
    ]
  }
  shared 中的字段被所有 vars 共享：
    所有变量的质量戳 "2" 都是 192
    所有变量的时间戳 "4" 都是 1712567890
    只有值 "3" 不同
  当设备正常运行时，大量变量的质量戳和时间戳相同，用共享值避免重复传输
```

回写指令也一样，不同网关接受的控制报文格式各不相同。

#### 设计方案：协议配置 + 解析器注册表

在 sys_gateway 表中新增协议相关字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| protocol_type | VARCHAR(32) NOT NULL | 协议类型标识（如 `flat_json`、`numeric_key`、`custom`） |
| push_mode | VARCHAR(16) NOT NULL DEFAULT 'full' | 推送模式：`full`（每次全量）/ `incremental`（首次全量+后续增量）/ `delta_only`（纯增量） |
| parse_config | JSONB | 上行解析配置（键映射、共享值规则、时间戳字段名、编码方式等） |
| write_config | JSONB | 下行回写配置（指令报文模板、topic 模板、QoS 等） |
| topic_pattern | VARCHAR(256) | 该网关的 MQTT topic 模式（如 `device/{gateway_id}/data`） |
| cmd_topic_pattern | VARCHAR(256) | 回写指令的 topic 模式（如 `device/{gateway_id}/cmd`） |

#### 统一中间结构 RawPoint

所有解析器最终输出统一的 RawPoint 列表，供变量处理引擎消费：

```java
public record RawPoint(
    String  varTag,      // 变量标识（对应 sys_variables.var_tag）
    Object  rawValue,    // 原始值（解析器提取后的值，尚未量程转换）
    int     quality,     // 质量戳（OPC 质量码：192=Good, 0=Bad, 64=Uncertain；无此字段时默认 192）
    Long    timestamp    // 时间戳（epoch_ms，无此字段时由系统填入接收时间）
) {}
```

#### 解析器注册表（上行）

使用策略模式，每种协议一个解析器实现：

```
接口: GatewayParser
  方法: List<RawPoint> parse(String gatewayId, byte[] payload, JSONB parseConfig)
  返回: List<RawPoint>

注册表: Map<String, GatewayParser> parserRegistry
  "flat_json"     → FlatJsonParser       （直接 $.key 取值）
  "nested_json"   → NestedJsonParser     （JSONPath 取值）
  "array_points"  → ArrayPointsParser    （遍历数组，按 tag 字段匹配）
  "numeric_key"   → NumericKeyParser     （数字键映射 + 共享值合并）
  "binary_base64" → BinaryBase64Parser   （Base64 解码 + 字节偏移提取）
  "custom"        → ScriptParser         （用户自定义脚本解析，GraalVM 沙箱执行）
```

#### parse_config 示例

```json
// flat_json 类型
{
  "timestamp_field": "ts",
  "timestamp_format": "epoch_ms",
  "value_mapping": {
    "temp": "temperature_01",
    "pressure": "pressure_01"
  }
}

// nested_json 类型
{
  "timestamp_path": "$.timestamp",
  "timestamp_format": "iso8601",
  "points": [
    { "json_path": "$.data.temperature", "var_tag": "temperature_01" },
    { "json_path": "$.data.pressure", "var_tag": "pressure_01" }
  ]
}

// numeric_key 类型（数字键映射 + 共享值）
{
  "key_mapping": {
    "1": "var_tag",
    "2": "quality",
    "3": "value",
    "4": "timestamp"
  },
  "vars_path": "$.vars",
  "shared_path": "$.shared",
  "timestamp_format": "epoch_s",
  "quality_mapping": {
    "192": "good",
    "0": "bad",
    "64": "uncertain"
  }
}
// 解析逻辑：
//   1. 先读 shared_path 下的共享字段，作为所有变量的默认值
//   2. 遍历 vars_path 数组中的每个对象
//   3. 对每个对象，按 key_mapping 翻译数字键为语义字段
//   4. 对象中缺失的字段从 shared 中补全（共享值兜底）
//   5. 输出统一 RawPoint

// custom 类型（用户脚本解析）
{
  "script_lang": "javascript",
  "script": "function parse(payload) { /* 用户自定义解析逻辑 */ return [...]; }"
}
```

#### 增量推送处理（push_mode = incremental / delta_only）

当网关配置为增量推送模式时，系统需要维护每个网关的变量快照：

```
ConcurrentHashMap<String, Map<String, RawPoint>> gatewaySnapshot
  key:   gateway_id
  value: 该网关所有变量的最后一次有效值 { varTag → RawPoint }
```

处理流程：

```
收到增量报文（只包含变化的变量）
  ↓
解析器正常解析，得到 List<RawPoint>（本次变化的变量）
  ↓
合并到 gatewaySnapshot：
  for each point in parsedPoints:
    gatewaySnapshot[gatewayId][point.varTag] = point
  ↓
投入变量处理引擎的数据 = parsedPoints（只处理变化的）
  未推送的变量 = 值没变，内存缓存中已有正确值，不需要重新处理
```

首次连接处理：

```
push_mode = incremental:
  首次连接时网关推送全量 → 全部解析并更新 gatewaySnapshot → 全部投入处理引擎
  后续只推变化量 → 增量更新 gatewaySnapshot → 只处理变化的

push_mode = delta_only:
  网关永远只推变化量
  服务启动时 gatewaySnapshot 为空
  → 第一批数据到达时逐步填充
  → 缺失的变量保持「值未知」状态，quality 标记为 uncertain
  → 直到收到该变量的第一次数据
```

网关重连处理：

```
检测到网关重新连接（MQTT session 重建）
  ↓
if push_mode = incremental:
  清空该网关的 gatewaySnapshot
  等待网关推送首次全量数据重建快照
if push_mode = delta_only:
  不清空快照（增量模式下快照是唯一的全量状态）
  标记所有变量 quality = uncertain，等新数据刷新
```

#### 编码器注册表（下行）

回写方向同样使用策略模式：

```
接口: GatewayEncoder
  方法: byte[] encode(String gatewayId, WriteCommand cmd, JSONB writeConfig)
  返回: 编码后的 MQTT payload

注册表: Map<String, GatewayEncoder> encoderRegistry
  "flat_json"     → FlatJsonEncoder      （直接 key-value）
  "nested_json"   → NestedJsonEncoder    （按模板填充）
  "binary"        → BinaryEncoder        （按字节偏移写入）
  "custom"        → ScriptEncoder        （用户自定义脚本编码）
```

#### write_config 示例

```json
{
  "cmd_topic": "device/{gateway_id}/cmd",
  "qos": 1,
  "template": {
    "action": "{command.action}",
    "target": "{command.tag}",
    "operator": "{command.operator}",
    "ts": "{command.ts}"
  }
}
```

#### 数据流走向

```
上行（采集）:
  MQTT 原始报文 → parserRegistry.get(gateway.protocolType).parse(...)
                → List<RawPoint>（统一格式）
                → 变量处理引擎逐条处理

下行（回写）:
  WriteCommand（统一格式） → encoderRegistry.get(gateway.protocolType).encode(...)
                          → byte[] payload
                          → MQTT Client 发布到 gateway.cmdTopicPattern
```

#### 新增网关协议的步骤

1. 实现 `GatewayParser` 和 `GatewayEncoder` 接口
2. 注册到 `parserRegistry` / `encoderRegistry`（可通过 Spring @Component 自动扫描，或 SPI 机制）
3. 在 sys_gateway 表中配置 `protocol_type` 和对应的 `parse_config` / `write_config`
4. 不需要改任何已有代码

对于非标协议，`custom` 类型允许用户在 `parse_config.script` 中写自定义脚本解析，在 GraalVM 沙箱中执行。

### 5.10 统一回写调度器（WriteDispatcher）

职责：提供一个统一的控制指令下发入口，所有需要「向设备写值」的场景都走这个模块，而不是各自直接操作 MQTT Client。

#### 为什么需要统一

当前有三个来源需要回写设备：

| 来源 | 场景 |
|------|------|
| WebSocket 控制指令 | 前端操作员点击按钮下发控制命令 |
| 用户自定义脚本 | 脚本中调用 `writeTag("valve_01", "open")` |
| 任务 Action 执行器 | action_type=2（MQTT 发布）中的回写动作 |

如果三个来源各自拼报文、各自发 MQTT，会导致：
- 协议编码逻辑重复散落在三处
- 权限校验逻辑重复散落在三处
- 回写审计日志口径不一致
- 后续加限流/排队/去重等策略要改三个地方

#### WriteDispatcher 接口

```java
public interface WriteDispatcher {

    /**
     * 下发控制指令（异步，返回 CompletableFuture）
     *
     * @param command  统一指令对象
     * @param source   来源标识（WS / SCRIPT / TASK）
     * @return 执行结果
     */
    CompletableFuture<WriteResult> dispatch(WriteCommand command, WriteSource source);
}
```

#### WriteCommand 统一指令对象

```java
public record WriteCommand(
    String  tag,         // 目标变量 tag（如 valve_01）
    Object  value,       // 写入的值（可以是数值、字符串、布尔）
    String  action,      // 动作（open / close / set 等，由设备协议定义）
    Long    operatorId,  // 操作人 user_id（脚本无操作人时为 null）
    String  operatorName,// 操作人名（审计用）
    long    ts           // 指令创建时间戳
) {}
```

#### 执行流水线

```
WriteCommand 进入 WriteDispatcher
  ↓
第 1 步：变量查找
  从 variableCache 中找到变量配置
  找不到 → 返回错误「变量不存在」
  ↓
第 2 步：权限校验
  来源 = WS   → 必须校验 operatorId 对该变量的写权限（sys_permissions）
  来源 = SCRIPT → 根据触发该脚本的任务配置决定是否校验（task.require_auth）
  来源 = TASK  → 系统内部调用，跳过权限校验
  ↓
第 3 步：读写模式检查
  变量 rw_mode 必须为 W（只写）或 RW（读写）
  rw_mode = R → 返回错误「该变量为只读」
  ↓
第 4 步：网关查找 + 协议编码
  根据变量配置找到归属网关 → 获取 gateway.protocolType + writeConfig
  调用 encoderRegistry.get(protocolType).encode(command, writeConfig)
  ↓
第 5 步：MQTT 发布
  发布到 gateway.cmdTopicPattern 解析后的实际 topic
  QoS 取 writeConfig 中配置的值（默认 1）
  ↓
第 6 步：等待回执（可选）
  如果 writeConfig.waitAck = true → 等待设备 ACK（超时 5 秒）
  否则 → 发布成功即视为完成
  ↓
第 7 步：审计日志
  写入审计记录：谁、什么时间、对哪个变量、下发了什么指令、来源、结果
  统一记录，不管来源是 WS 还是脚本还是任务
  ↓
第 8 步：返回结果
  WriteResult { success, message, ackTs }
```

#### 脚本沙箱中的回写 API

在 GraalVM 沙箱注入 `writeTag` 函数，内部调用 WriteDispatcher：

```javascript
// 用户脚本中可直接调用
writeTag("valve_01", "open");          // 简单动作
writeTag("setpoint_01", 85.5);         // 写数值
writeTag("pump_02", "close", { wait: true }); // 等待回执
```

沙箱注入实现：

```java
// 注入到 GraalVM Context 的宿主函数
context.getBindings("js").putMember("writeTag", (ProxyExecutable) args -> {
    String tag = args[0].asString();
    Object value = args[1].as(Object.class);
    WriteCommand cmd = new WriteCommand(tag, value, ...);
    // 通过 WriteDispatcher 统一下发，来源标记为 SCRIPT
    return writeDispatcher.dispatch(cmd, WriteSource.SCRIPT).join();
});
```

#### 与 WebSocket 控制指令的关系

原来 5.6 节中描述的「控制指令（前端 → 服务端）」流程简化为：

```
WebSocket 收到 command 消息
  → 反序列化为 WriteCommand
  → writeDispatcher.dispatch(command, WriteSource.WS)
  → 拿到 WriteResult
  → 组装 command_ack 推给前端
```

WebSocket 层不再直接操作 MQTT Client，只做消息收发和格式转换。

#### 与 Action 执行器的关系

原来 action_type=2（MQTT 发布）中的设备回写动作也走 WriteDispatcher：

```
Action 执行器收到 action_type=2
  → 从 action_config 中提取 tag、value、action
  → 组装 WriteCommand
  → writeDispatcher.dispatch(command, WriteSource.TASK)
```

普通的 MQTT 发布（非设备回写，如向第三方系统推消息）仍直接使用 MQTT Client，不经过 WriteDispatcher。

## 六、数据保序策略

### 6.1 保序原则

核心原则：同一个网关的数据从接收到处理到推送，全程走同一条路径，严格串行，绝不分叉。

不同网关之间的数据没有顺序关系，可以完全并行处理。

### 6.2 各环节保序措施

| 环节 | 保序策略 |
|------|---------|
| MQTT 接收 → 分区队列 | 按 gateway_id 哈希分区，同一网关永远进同一队列 |
| 分区队列 → 变量处理 | 每个队列单线��消费，严格 FIFO |
| 变量处理 → 任务判断 | 同步执行，在同一线程内完成，不异步投递 |
| 任务判断 → Action 执行 | 异步投递，但同一变量触发的 action 通过分区队列保序 |
| 变量处理 → 前端推送 | 消息携带时间戳，前端丢弃乱序数据 |
| 变量处理 → 历史存储 | 数据携带时间戳，存储层按 ts 写入 |

### 6.3 为什么 change_type 和 condition_expr 必须同步判断

以温度变量为例，如果值的变化序列是 50 → 80 → 50：
- 同步串行处理：先判断 50→80（升高，触发 INCREASE），再判断 80→50（降低，触发 DECREASE），正确
- 异步乱序处理：可能变成 50→50（无变化）和 ?→80（错误基准），判断完全错误

因此数据改变类型判断和条件表达式评估必须在数据处理的主线程内同步完成。

## 七、线程模型

```
线程/线程池           数量       职责                   保序要求
--------------------------------------------------------------------
MQTT-IO              1~2       接收 MQTT 消息          无
                               计算分区号投入队列

Partition-Workers     8~16      每个线程绑定一个分区      严格保序
                               串行处理变量逻辑         这是核心保序点

Timer-Scheduler       2~4       定时任务触发            无

Action-Executor       4~8       执行 HTTP/MQTT/DB/     同变量保序
                               脚本等慢操作

WebSocket-Pusher      2~4       攒批推送前端            带时间戳

Script-Sandbox        2~4       GraalVM 沙箱执行        同变量保序
                               用户脚本

Storage-Writer        2~4       批量写入 PostgreSQL     带时间戳
--------------------------------------------------------------------
总计                  21~42     个线程
```

如果使用 Java 21 虚拟线程，Action-Executor 和 Script-Sandbox 可以使用虚拟线程，不占用平台线程资源，并发能力大幅提升。

## 八、并发能力评估

### 8.1 数据接入能力

```
100 个网关，每个网关 50 个变量，每秒上报一次
= 5000 条消息/秒

8 个分区 Worker，每个 Worker 承担约 625 条/秒
单条处理耗时约 0.1ms（内存操作为主）
单个 Worker 理论吞吐 10000 条/秒
实际利用率约 6%，非常轻松
```

### 8.2 前端推送能力

```
1000 个 WebSocket 连接
单台 Java 服务轻松支撑数万连接
攒批推送：每 100ms 推一批，1 秒推 10 批
每批平均推送变化的变量数约 50~200 个
单机完全够用，不需要集群
```

5000 个变量，变化存储模式下，假设 30% 每秒有变化

PostgreSQL 分区表批量写入轻松支撑；启用 TimescaleDB 扩展后写入性能更优

## 九、高可用与容错

| 场景 | 应对方案 |
|------|---------|
| 脚本执行超时 | 100ms 超时强制终止，不影响主流程 |
| Action 执行失败 | 记录失败原因到 sys_tasks，支持重试策略 |
| MQTT 连接断开 | 自动重连，Broker 端配置持久会话，不丢消息 |
| WebSocket 连接断开 | 前端自动重连，重新订阅 |
| 某个分区 Worker 异常 | 异常捕获后继续处理下一条，不影响其他分区 |
| 时序数据库写入失败 | 数据暂存本地队列，待 PostgreSQL 恢复后补写 |
| 服务重启 | 从数据库重新加载所有配置，重建内存索引 |
| 服务启动 | store_on_startup = 1 的变量，首次收到数据无条件存储一次 |

## 十、WebSocket 通信协议

### 10.1 消息类型总览

| type 字段 | 方向 | 说明 |
|-----------|------|------|
| data | 服务端 → 前端 | 实时变量数据推送 |
| alarm | 服务端 → 前端 | 报警事件推送 |
| command | 前端 → 服务端 | 控制指令下发 |
| command_ack | 服务端 → 前端 | 控制指令执行回执 |
| subscribe | 前端 → 服务端 | 订阅变量 |
| unsubscribe | 前端 → 服务端 | 取消订阅 |

### 10.2 设计原则

- 一条 WebSocket 连接承载所有读写通信
- 所有消息统一 JSON 格式，通过 type 字段区分
- 控制指令必须有 ACK 回执，前端必须等待确认
- 所有消息携带时间戳 ts，用于保序和审计

系统运行期间，以下配置支持热更新，无需重启服务：

| 配置项 | 热更新方式 |
|--------|-----------|

| AI 服务配置（sys_ai_providers） | 修改后重新加载到内存，立即生效 |

## 十二、AI API 调用中心

前端低代码编辑器中的 AI 功能（代码生成、布局建议、脚本辅助等）需要调用大模型 API。前端不直接持有 API Key，所有 AI 请求统一经后端代理转发。

### 12.1 为什么不让前端直接调 AI API

- API Key 放前端 = 明文暴露，浏览器 DevTools 一看就知道
- 无法做并发控制，10 个用户同时发请求可能撞上游限流
- 无法统计用量、审计谁在用、花了多少 Token
- Key 轮换、失败重试、流式转发等逻辑放前端太重

### 12.2 sys_ai_providers 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL PK | 自增主键 |
| name | VARCHAR(64) NOT NULL | 显示名（如「GPT-4o」「DeepSeek-V3」「公司内网模型」） |
| provider_key | VARCHAR(64) UNIQUE NOT NULL | 唯一标识（如 `gpt4o`、`deepseek_v3`，前端选择时传这个） |
| base_url | VARCHAR(512) NOT NULL | API 端点（如 `https://api.openai.com/v1`） |
| api_key | VARCHAR(512) NOT NULL | API Key（AES 加密存储，不明文入库） |
| model | VARCHAR(128) NOT NULL | 默认模型名（如 `gpt-4o`、`deepseek-chat`） |
| max_concurrent | INT NOT NULL DEFAULT 5 | 该 Key 允许的最大并发请求数 |
| max_tokens_per_req | INT NOT NULL DEFAULT 4096 | 单次请求最大输出 Token 数 |
| enabled | BOOLEAN NOT NULL DEFAULT true | 是否启用 |
| sort_order | INT NOT NULL DEFAULT 0 | 前端下拉列表排序 |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 创建时间 |
| updated_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 最后修改时间 |

说明：
- 所有 provider 统一采用 **OpenAI 兼容格式**（`/v1/chat/completions`），不同厂商只是 base_url 和 model 不同
- api_key 字段 AES-256-GCM 加密存储，密钥放在应用配置文件中，不入库
- 管理接口返回时 api_key 只展示后 4 位，不返回完整值

### 12.3 sys_ai_usage_log 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL PK | 自增主键 |
| user_id | BIGINT NOT NULL | 调用者 user_id |
| provider_id | BIGINT NOT NULL | 使用的 AI 服务配置 ID |
| model | VARCHAR(128) | 实际使用的模型名 |
| prompt_tokens | INT | 输入 Token 数 |
| completion_tokens | INT | 输出 Token 数 |
| total_tokens | INT | 总 Token 数 |
| duration_ms | INT | 请求耗时（毫秒） |
| status | VARCHAR(16) NOT NULL | success / error / timeout |
| error_msg | TEXT | 失败时的错误信息 |
| scene | VARCHAR(32) | 调用场景（code_gen / layout_suggest / script_assist 等） |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | 调用时间 |

索引：

```sql
CREATE INDEX idx_ai_log_user ON sys_ai_usage_log(user_id, created_at DESC);
CREATE INDEX idx_ai_log_provider ON sys_ai_usage_log(provider_id, created_at DESC);
```

### 12.4 AI 代理服务架构

```
前端编辑器（请求 AI 生成代码 / 布局建议）
  ↓
POST /api/ai/chat/completions
  Header: Authorization: Bearer <用户登录 Token>
  Body: { provider_key: "gpt4o", messages: [...], stream: true }
  ↓
[后端 AI 代理服务]
  ↓
  1. 身份验证（JWT 校验）
  2. 查找 provider_key 对应的 sys_ai_providers 配置
  3. 并发控制（Semaphore 限流）
  4. 解密 api_key，组装 OpenAI 格式请求
  5. 转发到上游 AI 服务
  6. 流式转发（SSE）回前端
  7. 请求结束后异步写入 sys_ai_usage_log
```

### 12.5 并发控制

每个 provider 用一个独立的 Semaphore 控制并发：

```java
ConcurrentHashMap<String, Semaphore> providerSemaphores
  key:   provider_key
  value: Semaphore(maxConcurrent)

// 调用流程
1. semaphore.tryAcquire(3, TimeUnit.SECONDS)
     成功 → 继续
     失败 → 返回 429 Too Many Requests，前端提示「当前使用人数过多，请稍后再试」
2. try { 转发请求 } finally { semaphore.release() }
```

多个 Key 的并发弼此独立。如果配置了 3 个 Key（maxConcurrent 分别为 5、3、2），系统总并发能力 = 5 + 3 + 2 = 10。

### 12.6 前端交互流程

```
1. 前端调用 GET /api/ai/providers 获取可用服务列表
   返回: [{ providerKey: "gpt4o", name: "GPT-4o", model: "gpt-4o" }, ...]
   （只返回 enabled=true 的，不返回 api_key）

2. 用户在编辑器中选择一个 AI 服务（下拉菜单）
   选择保存在前端 localStorage，下次打开自动恢复

3. 用户触发 AI 生成（如点击「AI 生成代码」按钮）
   前端发送 POST /api/ai/chat/completions
   Body: {
     "provider_key": "gpt4o",
     "messages": [
       { "role": "system", "content": "你是一个 SCADA 脚本开发助手..." },
       { "role": "user", "content": "帮我写一个温度超限报警脚本" }
     ],
     "stream": true
   }

4. 后端流式返回（SSE），前端逐字渲染到 Monaco Editor
```

### 12.7 流式转发（SSE Proxy）

前端请求 stream=true 时，后端做 SSE 透明代理：

```
前端 ← SSE ← 后端代理 ← SSE ← 上游 AI 服务
```

- 后端用 Java HttpClient 异步读取上游的 SSE 流
- 每收到一个 chunk 立即写入前端响应流，不缓冲
- 请求结束后从最后一个 chunk 的 usage 字段提取 token 用量，异步写入 usage_log
- 前端断开连接时，后端同步取消上游请求，不浪费 Token

非流式请求（stream=false）：后端等待上游完整响应后一次性返回。

### 12.8 安全设计

| 层面 | 措施 |
|------|------|
| Key 存储 | AES-256-GCM 加密存储，加密密钥放应用配置文件，不入库 |
| Key 展示 | 管理接口只返回 `sk-****xxxx`，不返回完整 Key |
| 身份验证 | 前端必须携带登录 Token，未登录 → 401 |
| 权限控制 | 普通用户只能调用 AI，只有 admin 角色能管理 provider 配置 |
| 请求限流 | 每个 provider 独立 Semaphore 限流，超出返回 429 |
| 输入清洗 | 前端传入的 messages 不允许包含 system prompt 注入，system prompt 由后端拼接 |
| 超时控制 | 单次 AI 请求最长 60 秒，超时强制断开上游连接 |
| 审计追踪 | 每次调用记录 user_id + provider_id + token 用量 + 场景 |

## 十三、部署架构

### 单机部署（推荐初期）

```
单台服务器（4核 8G 以上）
  ├── Java 应用（Spring Boot）
  ├── MQTT Broker（EMQX / Mosquitto）
  └── PostgreSQL（系统表 + 时序数据，统一存储）
```

100 个网关 + 1000 个前端用户，单机完全够用。

### 扩展部署（未来）

如果需要扩展到更大规模：
- Java 应用可水平扩展，通过 Redis 共享变量缓存
- MQTT Broker 可集群部署
- PostgreSQL 可主从复制或读写分离，时序数据量大时可启用 TimescaleDB 扩展或独立部署时序库
- WebSocket 连接通过 Nginx 负载均衡分发
